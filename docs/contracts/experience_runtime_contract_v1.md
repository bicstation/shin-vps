# Purpose

This document defines the constitutional Runtime Contract for all Experience APIs within SHIN CORE LINX.

The Experience Runtime separates semantic meaning, user presentation, search-engine metadata, and reality data into independent Runtime layers.

This separation preserves Backend Authority while allowing the Frontend Experience Layer to render a consistent user experience without semantic interpretation.

---

# Constitutional Runtime Flow

```text
Reality

↓

Meaning

↓

Presentation

↓

SEO

↓

Experience
```

Each Runtime layer owns a single responsibility.

---

# Runtime Structure

The canonical Experience Runtime SHALL follow the structure below.

```text
meaning

presentation

seo

data
```

---

# Meaning Runtime

## Purpose

Meaning Runtime defines the semantic identity of an Experience.

Meaning is the constitutional semantic authority.

Meaning SHALL NOT be generated by the Frontend.

---

## Example Fields

```text
identity

mission

user_intent

meaning_statement

existence_reason
```

---

## Responsibility

Meaning Runtime answers.

```text
Why does this page exist?

What semantic role does it perform?

What user intent does it satisfy?
```

---

# Presentation Runtime

## Purpose

Presentation Runtime defines the recommended user-facing presentation.

Presentation SHALL NOT redefine semantic meaning.

Presentation projects Runtime Meaning into a human-friendly experience.

---

## Runtime Contract

```text
presentation

    title

    subtitle

    description
```

---

## Responsibility

Presentation Runtime answers.

```text
How should this page be presented?

What should users see first?

How should the Experience begin?
```

---

## Example

```json
{
    "presentation": {

        "title":
            "PC商品一覧",

        "subtitle":
            "公開中のPCを一覧から探す",

        "description":
            "AI・ゲーミング・ビジネスなど用途に合わせて公開中のPCを一覧から探索できます。"
    }
}
```

---

# SEO Runtime

## Purpose

SEO Runtime defines metadata for search engines.

SEO metadata is independent from Presentation Runtime.

---

## Minimum Contract

```text
seo.title

seo.description

seo.keywords

seo.canonical
```

---

## Responsibility

SEO Runtime answers.

```text
How should search engines understand this page?
```

---

# Reality Data Runtime

## Purpose

Reality Data contains the actual business data consumed by the Experience.

Examples.

```text
products

discover

ranking

finder

related_products
```

Reality Data SHALL remain independent from Meaning, Presentation, and SEO.

---

# Frontend Experience Contract

The Frontend Experience Layer SHALL consume Runtime information in the following order.

```text
presentation

↓

seo

↓

meaning
```

Preferred rendering.

```text
Hero Title

↓

presentation.title
```

```text
Hero Subtitle

↓

presentation.subtitle
```

```text
Hero Description

↓

presentation.description
```

Development fallback only.

```text
presentation.title

↓

seo.title

↓

meaning.identity
```

Meaning Runtime SHALL NOT be used as presentation text in production.

---

# Authority Boundaries

## Backend Authority

Responsible for.

```text
Reality

Meaning

Presentation

SEO
```

The Backend defines Runtime Authority.

---

## Adapter Authority

Responsible for.

```text
Transport

Normalize

Projection
```

The Adapter SHALL NOT generate Meaning or Presentation.

---

## Frontend Experience

Responsible for.

```text
Render

Layout

Navigation

Interaction
```

The Frontend SHALL consume Runtime information.

The Frontend SHALL NOT generate Runtime Authority.

Presentation Runtime may be extended without breaking existing clients.

Examples.

```text
hero_image

hero_layout

cta

badge

theme

icon
```

These fields belong to Presentation Runtime and SHALL NOT affect Meaning Runtime.
